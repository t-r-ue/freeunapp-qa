# Product Requirements Document (PRD)
## Embedded Web Feedback Widget (Version 2)

---

## 1. Executive Summary & Goal
The goal of this widget is to allow real website users on FreeUnApp to share suggestions, bug reports, and feedback directly from their browsers without installing extensions or leaving the page. 

It aims to replace heavy, disruptive feedback questionnaires with a lightweight, 3-second micro-interaction. The widget automatically captures technical diagnostics (current page URL, screen specifications, browser information, console errors, and page screenshots) and uploads them directly to the QA Feedback Dashboard.

---

## 2. User Experience & Design Guardrails
To prevent the feedback widget from acting like intrusive pop-ups or ads, the following design guidelines are enforced:

### A. Passive Docking (Default State)
* **Visual**: A tiny, elegant vertical button docked on the right border of the browser window (e.g. `💬 Feedback`).
* **Design**: Uses square edges (matching the 2-3px radius styling of the dashboard) and subtle warm zinc colors to merge cleanly with the host site.
* **Behavior**: It remains silent and stationary, never overlapping main buttons or input fields.

### B. Proactive Micro-Nudge (Optional)
Instead of blocking the browser window, proactive nudges are delivered as non-blocking toasts in the bottom-right corner:
* **No Backdrop**: Does not grey out the screen. The user can continue clicking and browsing without interacting with the nudge.
* **Auto-Dismiss**: If ignored, the nudge slides out of view after 7 seconds.

### C. Strict Anti-Annoyance Memory
* The moment a user interacts with a nudge (closes it, types a comment, or dismisses it), the widget sets a key `freeunapp_feedback_dismissed = true` in the browser's `localStorage`.
* Once set, the widget **remains completely silent** and will never nudge that user again.

---

## 3. Feedback Submission Flow
When a user clicks the tab or micro-nudge, a clean slide-out card opens:

```
┌──────────────────────────────────────────────┐
│  How was your experience?                    │
│  😠  😐  😊                                  │
│                                              │
│  Choose category:                            │
│  [Bug]  [Suggestion]  [Question]  [Compliment]│
│                                              │
│  Tell us more...                             │
│  ┌────────────────────────────────────────┐  │
│  │                                        │  │
│  └────────────────────────────────────────┘  │
│  [x] Include screenshot of this page         │
│                                              │
│                      [ Cancel ] [ Send ]     │
└──────────────────────────────────────────────┘
```

1. **Step 1: Emoji Rating**: Tap an emoji (😠 = Bug/Severe, 😐 = Medium, 😊 = Positive).
2. **Step 2: Intent Categorization**: Select category (Bug, Suggestion, Question, or Compliment).
3. **Step 3: Text Input**: Optional text box for quick descriptions.
4. **Step 4: Auto Screenshot Option**: A pre-checked box allowing them to include their current screen visual.
5. **Submission**: Upon clicking "Send", the card shows a subtle loading state, triggers a success toast, and slides shut.

---

## 4. Diagnostics Harvested (The "Heavy Lifting")
While the user only types a brief comment, the widget automatically packages and attaches the following data:
* **Active Route/URL**: Tells developers exactly which page the user was looking at.
* **Screen Specifications**: Viewport height, width, and device pixel ratio.
* **User Agent Info**: Browser engine, version, and operating system.
* **Console Error Cache**: Dynamically intercepts the last 5 `console.error` logs generated by the website.
* **HTML5 Canvas Screenshot**: Renders the visible webpage layout directly into a static image file.

---

## 5. Integration with the QA Dashboard
All submissions are sent directly to the QA serverless API (`POST /api/issues`). They are automatically parsed and formatted for the developer team:
* **Title Format**: `[User Feedback] {Category} - Rating: {Emoji}`
* **Assigned Severity**: 😠 maps to `Severe`, 😐 maps to `Medium`, 😊 maps to `Pass`.
* **Issue Description**: Contains the user's text comment plus the formatted diagnostic metadata and browser console logs.
* **Screenshots**: Linked directly to the detail view as a thumb.
* **Category Filtering**: Developers can toggle a sidebar pill or drop-down to filter between **Developer QA Audits** and **Real User Feedback** streams.
